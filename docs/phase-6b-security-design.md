# Phase 6B Security Design — Browser-Native Fingerprint Approvals

## Goal

Move `approve_plan` and `approve_merge` out of the CLI and into the dashboard
approval modal, while preserving the content-integrity and human-confirmation
guarantees the CLI fingerprint flow provides today. `merge_run` stays CLI-only
in this phase.

---

## Current CLI Security Model (What We Must Preserve)

1. **Content integrity** — `approvalBinding()` (`src/application/integrity.ts`)
   computes a deterministic SHA-256 hash of exactly what is being approved:
   plan config + task scopes for plan stage; commit SHAs + diff hashes for merge
   stage. This hash is the fingerprint.

2. **Human confirmation** — the CLI prints the fingerprint and requires the
   operator to type it back. Automation cannot satisfy this without knowing the
   hash in advance.

3. **One-time ticket** — `createActionTicket` / `consumeActionTicket`
   (`src/persistence/store.ts`). Tickets are bound to `(run_id, action,
   binding_hash, run_version)` and expire after a short window. Replay is
   impossible.

4. **Version lock** — `run_version` in the ticket means the ticket becomes
   invalid if the run state changes between display and consumption.

---

## Threat Model

| Threat | Mitigated by |
|---|---|
| Automated script submits approval | Typed hash prefix — cannot be computed without reading the server response |
| CSRF triggers approval POST | Existing localhost-only CORS check + `idempotency-key` header requirement |
| Replay of a prior approval | Action ticket is one-time-use; `consumed_at` set atomically |
| Run state changes between display and click | `run_version` in ticket; server re-checks before executing |
| Brute-forcing the hash prefix | Rate limit failed attempts (see §4) |

Note: a malicious manager proposal is not a meaningful threat here. Proposals
are generated by the operator's own manager in response to their own messages,
action and run/task IDs are validated server-side, and `commandCli`/`commandJson`
are always server-synthesised. The modal shows server-computed run state for
clarity, not as a security measure.

---

## Auth: Session Is Sufficient

The service binds only to `127.0.0.1`. Anyone who can reach the dashboard
already has local machine access and therefore terminal access. The gap between
"can use browser" and "can use terminal" does not exist for this tool, so there
is no need to require bearer token auth or a terminal-side challenge code for
approval routes. A valid `duet_session` cookie is sufficient — the same auth
level used for all other dashboard mutations.

## What Needs to Be Built

### 1. Approval API route (new)

```
POST /api/v1/runs/{runId}/approve
  body: { stage: "plan"|"merge", bindingHash: string, runVersion: number, confirm: string }
  auth: bearer OR session

POST /api/v1/runs/{runId}/merge
  stays CLI-only in Phase 6B
```

Server-side flow on `approve`:
1. Load run + tasks, compute `approvalBinding(run, tasks, stage)`.
2. Compare against client-supplied `bindingHash` — reject if mismatch.
3. Verify `runVersion` matches current `run.version`.
4. Verify `confirm` is the first 8 characters of the binding hash.
5. Create and consume an action ticket atomically (reuse existing
   `createActionTicket` / `consumeActionTicket` from `src/persistence/store.ts`).
6. Write the approval record and emit the approval event.

### 2. Binding hash preview route (new)

```
GET /api/v1/runs/{runId}/approval-preview?stage=plan|merge
  auth: bearer OR session
  returns: { bindingHash, stage, run: { id, goal, baseBranch, baseCommit },
             tasks?: [{ id, allowedPaths }],          // plan stage
             integration?: { commit, taskDiffHashes } // merge stage
           }
```

The modal fetches this on open. It gives the user the server-computed hash
and enough context to understand what they are approving.

### 3. Fingerprint modal UI (`src/dashboard/assets.ts`)

Triggered when a `approve_plan` or `approve_merge` fingerprint proposal card
is inspected. Flow:

1. Fetch `GET /approval-preview` on modal open.
2. Show: stage label, run ID/goal/branch/commit, full binding hash in a
   monospace block.
3. Typed confirmation field: user must type the first 8 characters of the hash.
4. Submit button enabled only when prefix matches.
5. On submit: POST to `/approve` with `bindingHash`, `runVersion`, `confirm`.
6. On success: close modal, refresh conversation and run state.

### 4. Rate limiting failed attempts

In-memory counter keyed by `runId + stage`, reset on success. Block further
attempts for 60 seconds after three consecutive failures. Prevents prefix
brute-force even though the hash is shown to the user (defends against
automated scripts that read the DOM and retry).

## What Stays CLI-Only in Phase 6B

| Action | Reason |
|---|---|
| `merge_run` | Highest consequence; irreversible without a git revert |
| CLI approval flow | Unchanged — `duet approve` must continue to work |
